# pie-project: Research Note

**Source**: https://github.com/pie-project/pie (147 stars, pre-release)
**Papers**: SOSP 2025 "Pie: A Programmable Serving System for Emerging LLM Applications" (Gim, Ma, Lee, Zhong — Yale); HotOS 2025 "Serve Programs, Not Prompts" (Gim, Zhong)
**Date of this note**: 2026-04-21

## What it is (one sentence)

Pie is a **programmable LLM serving system** that replaces the monolithic prefill-decode loop of vLLM / SGLang / TGI with **fine-grained API handlers** (embed, forward, sample, alloc_kvpage, …) orchestrated by user-provided **inferlets** — sandboxed WebAssembly programs that define the generation logic from the application side.

## Architecture in one paragraph

Three layers. **Application layer** runs user-written inferlets inside a Wasmtime sandbox; inferlets issue API calls to manipulate contexts (fill / flush / generate / fork), KV pages (alloc / copy / mask / export), and the model forward pass. **Control layer** (Rust, PyO3) virtualises KV pages and embeddings, schedules batches across concurrent inferlets using a work-conserving adaptive policy, and performs IPC to workers via ZeroMQ. **Inference layer** (Python `pie_worker`, PyTorch + FlashInfer CUDA kernels or custom Metal kernels) executes batched API calls on the accelerator. 42 APIs total, 18 core to the LLM forward/resource-management path. Inferlets can be written in Rust, C++, Go, Python, or JavaScript and compile to Wasm.

## Thesis (paraphrased, then quoted)

LLM serving is an OS problem: the serving system should be the OS, inferlets are the processes, KV pages are the pages, and the application — not the system — should drive cache and decoding policy. Quote from the HotOS paper: *"we propose that LLM serving system should transition from processing mere prompts to handling programs, which we term LLM Inference Programs (LIPs)."* Quote from the SOSP abstract: *"Pie decomposes the traditional generation loop into fine-grained service handlers exposed via an API and delegates control of the generation process to user-provided programs, called inferlets."*

## Hardware targets

NVIDIA (CUDA 12.6 / 12.8 with FlashInfer kernels) and Apple Silicon (Metal, via a custom `flashinfer_metal` kernel set). No Ascend, no AMD, no TPU. Benchmarks in the SOSP paper run on L40 and 8B-class models (Llama 3.2 1B cited explicitly).

## Published performance numbers

Measured vs vLLM, SGLang, LMQL, StreamingLLM on Llama 3.2 1B / L40:

- Standard text completion: **matches vLLM within 1.5–11% TPOT overhead** (2.4% at 8B, 11.4% at 1B).
- Agentic workflows: **1.1–2.4× lower latency, 1.3–3.4× higher throughput**.
- Graph-of-Thought / Recursion-of-Thought: up to 34% throughput lift over baseline vLLM workflows, 28% latency cut.
- Attention sink vs hand-built StreamingLLM: **1.5× lower latency, 30× higher throughput**.
- Adaptive batch scheduling: **17× higher throughput than eager (no batching)**, 8–40% over K-only / T-only policies.
- Inferlet cold-start: 35–81 ms; warm start: 10–50 ms. API call overhead: <30 μs (control layer), 10–300 μs (inference layer).

## Maturity

Pre-release research prototype (v0.2.5). Apache-2.0. SOSP 2025 artifacts-evaluated (functional + available + reproducible). Authors flag the scheduler as a single-node centralised bottleneck; scaling to multi-node is listed as future work alongside security hardening (inferlets have wide syscall surface).

## Not about AI coding agents

Despite the `.agent/skills/pie/SKILL.md` file and `AGENTS.md` in the repo — which are an agent *runbook* for contributors using Claude Code / Cursor to work on Pie, not a claim about agent-generated inference — Pie is neither generated by nor built for AI coding agents. It is a *research system for agentic LLM applications* (tree-of-thought, speculative decoding, multi-agent pipelines), which is a different thing.
